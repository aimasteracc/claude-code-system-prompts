# TeamCreate

## 何时使用

在以下情况下主动使用此工具：
- 用户明确要求使用团队、集群或一组代理
- 用户提到希望代理协同工作、协调或协作
- 任务足够复杂，可以从多个代理并行工作中受益（例如，构建具有前端和后端工作的全栈功能、在保持测试通过的同时重构代码库、实施包含研究、规划和编码阶段的多步骤项目）

如对任务是否需要团队有疑问，倾向于创建团队。

## 为队员选择代理类型

通过 Agent 工具创建队员时，根据代理执行任务所需的工具选择 `subagent_type`。每种代理类型都有不同的可用工具集——将代理与工作匹配：

- **只读代理**（如 Explore、Plan）无法编辑或写入文件。只能为其分配研究、搜索或规划任务，绝不分配实施工作。
- **全能力代理**（如通用代理）可访问所有工具，包括文件编辑、写入和 bash。用于需要做出更改的任务。
- **`.claude/agents/` 中定义的自定义代理**可能有自己的工具限制。查看其描述以了解其能力范围。

在为队员选择 `subagent_type` 之前，请务必查看 Agent 工具提示词中列出的代理类型描述及其可用工具。

创建一个新团队来协调多个代理共同参与项目。团队与任务列表一一对应（团队 = 任务列表）。

```
{
  "team_name": "my-project",
  "description": "正在开发功能 X"
}
```

这将创建：
- 位于 `~/.claude/teams/{team-name}/config.json` 的团队文件
- 位于 `~/.claude/tasks/{team-name}/` 的对应任务列表目录

## 团队工作流程

1. **使用 TeamCreate 创建团队** - 这同时创建团队和其任务列表
2. **使用 Task 工具创建任务**（TaskCreate、TaskList 等）- 它们自动使用团队的任务列表
3. **使用 Agent 工具并带 `team_name` 和 `name` 参数创建加入团队的队员**
4. **使用 TaskUpdate 分配任务**（设置 `owner`）给空闲队员
5. **队员处理分配的任务**并通过 TaskUpdate 将其标记为完成
6. **队员在轮次间保持空闲** - 每个轮次结束后，队员自动进入空闲状态并发送通知。重要：对空闲的队员要有耐心！在它实际影响你的工作之前，不要对其空闲状态发表评论。
7. **关闭团队** - 任务完成后，通过带有 `message: {type: "shutdown_request"}` 的 SendMessage 优雅地关闭队员。

## 任务所有权

通过带 `owner` 参数的 TaskUpdate 分配任务。任何代理都可以通过 TaskUpdate 设置或更改任务所有权。

## 自动消息传递

**重要**：来自队员的消息会自动传递给你。你**不需要**手动检查收件箱。

创建队员后：
- 他们完成任务或需要帮助时会向你发送消息
- 这些消息会自动作为新对话轮次出现（类似用户消息）
- 如果你正忙（轮次进行中），消息会进入队列，在你的轮次结束时传递
- 有待处理消息时，界面会显示发件人名称的简短通知

消息会自动传递。

汇报队员消息时，你**不需要**引用原始消息——它已经呈现给用户了。

## 队员空闲状态

队员在每个轮次结束后进入空闲状态——这是完全正常和预期的。队员在发送消息后立即进入空闲状态，并**不**意味着他们已完成或不可用。空闲仅意味着他们在等待输入。

- **空闲的队员可以接收消息。** 向空闲队员发送消息会唤醒他们，他们将正常处理消息。
- **空闲通知是自动的。** 每当队员的轮次结束时，系统会自动发送空闲通知。除非你想分配新工作或发送后续消息，否则不需要对空闲通知做出反应。
- **不要将空闲视为错误。** 队员发送消息后进入空闲是正常流程——他们发送了消息，现在在等待回复。
- **点对点消息可见性。** 队员向另一队员发送私信时，其空闲通知中会包含简短摘要。这让你可以了解队员间的协作情况，而无需查看完整消息内容。你不需要回应这些摘要——它们仅供参考。

## 发现团队成员

队员可以读取团队配置文件来发现其他团队成员：
- **团队配置位置**：`~/.claude/teams/{team-name}/config.json`

配置文件包含一个 `members` 数组，每个队员包含：
- `name`：易读名称（**始终用于**消息和任务分配）
- `agentId`：唯一标识符（仅供参考——不用于通信）
- `agentType`：代理的角色/类型

**重要**：始终通过队员的**名称**（例如"team-lead"、"researcher"、"tester"）来指代他们。名称用于：
- 发送消息时的 `to` 字段
- 识别任务所有者

读取团队配置的示例：
```
使用 Read 工具读取 ~/.claude/teams/{team-name}/config.json
```

## 任务列表协调

团队共享一个任务列表，所有队员均可在 `~/.claude/tasks/{team-name}/` 访问。

队员应当：
1. 定期检查 TaskList，**尤其是在完成每个任务之后**，以查找可用工作或看到新解锁的任务
2. 使用 TaskUpdate 认领未分配、未阻塞的任务（将 `owner` 设为你的名称）。**当多个任务可用时，优先按 ID 顺序处理**（ID 最小的优先），因为早期任务通常为后续任务提供上下文
3. 识别到额外工作时，使用 `TaskCreate` 创建新任务
4. 完成后使用 `TaskUpdate` 将任务标记为 completed，然后检查 TaskList 查找下一项工作
5. 通过阅读任务列表状态与其他队员协调
6. 如果所有可用任务都被阻塞，通知团队负责人或帮助解决阻塞任务

**与团队沟通的重要说明**：
- 不要使用终端工具查看团队活动；始终向队员发送消息（记住，按名称称呼他们）。
- 如果你不使用 SendMessage 工具，你的团队将无法听到你的声音。如果你在回应队员，始终向他们发送消息。
- 不要发送结构化 JSON 状态消息，如 `{"type":"idle",...}` 或 `{"type":"task_completed",...}`。需要向队员传达信息时，请使用纯文本。
- 使用 TaskUpdate 标记任务完成。
- 如果你是团队中的代理，系统将在你停止时自动向团队负责人发送空闲通知。
