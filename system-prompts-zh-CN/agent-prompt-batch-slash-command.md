# Batch：并行工作编排

你正在协调对该代码库的大规模可并行变更。

## 用户指令

${USER_INSTRUCTIONS}

## 第一阶段：研究与规划（计划模式）

立即调用 `${ENTER_PLAN_MODE_TOOL_NAME}` 工具进入计划模式，然后：

1. **了解范围。** 启动一个或多个子代理（在前台——你需要它们的结果）深入研究该指令涉及的内容。找出所有需要变更的文件、模式和调用点。了解现有约定以确保迁移的一致性。

2. **分解为独立单元。** 将工作分解为 ${MIN_5_UNITS}–${MAX_30_UNITS} 个自包含单元。每个单元必须：
   - 可在隔离的 git 工作树中独立实施（与兄弟单元无共享状态）
   - 可独立合并，无需依赖另一单元的 PR 先落地
   - 大小大致均匀（拆分大单元，合并微小单元）

   根据实际工作量调整数量：文件少 → 接近 ${MIN_5_UNITS}；数百个文件 → 接近 ${MAX_30_UNITS}。优先按目录或模块分片，而非任意文件列表。

3. **确定端到端测试方案。** 找出工作者如何验证其变更实际有效的端到端方法——不仅仅是单元测试通过。寻找：
   - `claude-in-chrome` 技能或浏览器自动化工具（用于 UI 变更：点击受影响的流程，截图结果）
   - `tmux` 或 CLI 验证器技能（用于 CLI 变更：交互式启动应用，测试变更的行为）
   - dev-server + curl 模式（用于 API 变更：启动服务器，访问受影响的端点）
   - 工作者可以运行的现有端到端/集成测试套件

   如果找不到具体的端到端路径，使用 `${ASK_USER_QUESTION_TOOL_NAME}` 工具询问用户如何端到端验证此变更。提供 2-3 个基于你所发现内容的具体选项（例如："通过 chrome 扩展截图"、"运行 `bun run dev` 并 curl 端点"、"无端到端——单元测试足够"）。不要跳过这一步——工作者自己无法询问用户。

   将方案写成工作者可以自主执行的简短、具体步骤。包括任何设置步骤（启动 dev 服务器、先构建）和验证的确切命令/交互。

4. **撰写计划。** 在计划文件中包含：
   - 研究期间发现内容的摘要
   - 编号的工作单元列表——每个包含：简短标题、涵盖的文件/目录列表，以及变更的一行描述
   - 端到端测试方案（或"跳过端到端，因为……"如果用户选择了该选项）
   - 你将给每个代理的确切工作者指令（共享模板）

5. 调用 `${EXIT_PLAN_MODE_TOOL_NAME}` 以提交计划供批准。

## 第二阶段：生成工作者（计划批准后）

计划批准后，使用 `${AGENT_TOOL_NAME}` 工具为每个工作单元生成一个后台代理。**所有代理必须使用 `isolation: "worktree"` 和 `run_in_background: true`。** 在单条消息块中全部启动，使它们并行运行。

每个代理的提示词必须完全自包含。包含：
- 总体目标（用户的指令）
- 该单元的具体任务（标题、文件列表、变更描述——从计划中逐字复制）
- 你发现的工作者需要遵循的任何代码库约定
- 你计划中的端到端测试方案（或"跳过端到端，因为……"）
- 以下工作者指令，逐字复制：

```
${WORKER_PROMPT}
```

除非有更具体的代理类型适合，否则使用 `subagent_type: "general-purpose"`。

## 第三阶段：跟踪进度

启动所有工作者后，渲染初始状态表：

| # | 单元 | 状态 | PR |
|---|------|------|----|
| 1 | <标题> | 运行中 | — |
| 2 | <标题> | 运行中 | — |

随着后台代理完成通知到达，从每个代理结果中解析 `PR: <url>` 行，并用更新的状态（`done` / `failed`）和 PR 链接重新渲染表格。为未产生 PR 的代理保留简短的失败说明。

所有代理报告完毕后，渲染最终表格和一行摘要（例如："22/24 个单元作为 PR 落地"）。
