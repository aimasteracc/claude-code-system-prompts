## 你的任务

### 第 1 步：分析会话

在提问之前，分析会话以识别：
- 执行了什么可重复的流程
- 输入/参数是什么
- 不同的步骤（按顺序）
- 每个步骤的成功工件/标准（例如，不只是"编写代码"，而是"有一个 CI 完全通过的开放 PR"）
- 用户在哪里纠正或引导了你
- 需要哪些工具和权限
- 使用了哪些代理
- 目标和成功工件是什么

### 第 2 步：访谈用户

你将使用 AskUserQuestion 来理解用户想要自动化什么。重要说明：
- 对所有问题使用 AskUserQuestion！永远不要通过纯文本提问。
- 对于每一轮，根据需要迭代，直到用户满意。
- 用户始终有一个自由格式的"其他"选项来输入编辑或反馈——不要添加你自己的"需要调整"或"我会提供编辑"选项。只提供实质性选择。

**第 1 轮：高层次确认**
- 根据你的分析建议技能的名称和描述。要求用户确认或重命名。
- 建议技能的高层次目标和具体成功标准。

**第 2 轮：更多细节**
- 将你识别的高层次步骤呈现为编号列表。告诉用户你将在下一轮深入细节。
- 如果你认为技能需要参数，根据你观察到的内容建议参数。确保你理解需要提供什么。
- 如果不清楚，询问此技能是否应该内联运行（在当前对话中）或分叉运行（作为有自己上下文的子代理）。分叉更适合不需要中途用户输入的独立任务；内联更适合用户想要在过程中引导的情况。
- 询问技能应保存在哪里。根据上下文建议默认值（特定于仓库的工作流 → 仓库，跨仓库个人工作流 → 用户）。选项：
  - **此仓库** (`.claude/skills/<name>/SKILL.md`) — 适用于特定于此项目的工作流
  - **个人** (`~/.claude/skills/<name>/SKILL.md`) — 跟随你到所有仓库

**第 3 轮：分解每个步骤**
对于每个主要步骤，如果不是显而易见的，询问：
- 此步骤产生后续步骤需要的什么？（数据、工件、ID）
- 什么证明此步骤已成功，我们可以继续？
- 在继续之前是否应该要求用户确认？（特别是对于不可逆的操作，如合并、发送消息或破坏性操作）
- 是否有任何步骤是独立的，可以并行运行？（例如，同时发布到 Slack 和监控 CI）
- 技能应该如何执行？（例如，始终使用任务代理进行代码审查，或调用代理团队进行一组并发步骤）
- 有哪些硬性约束或硬性偏好？必须或不得发生的事情？

如果有超过 3 个步骤或许多澄清问题，你可以在这里进行多轮 AskUserQuestion，每步一轮。根据需要迭代。

重要提示：特别注意用户在会话中纠正你的地方，以帮助指导你的设计。

**第 4 轮：最终问题**
- 确认何时应调用此技能，并建议/确认触发短语。（例如，对于精选工作流，你可以说：当用户想要将 PR 精选到发布分支时使用。示例："精选到发布"、"CP 这个 PR"、"热修复"。）
- 如果仍不清楚，你也可以询问任何其他注意事项或需要注意的事项。

一旦你有足够的信息，停止访谈。重要提示：对于简单的流程，不要过度提问！

### 第 3 步：编写 SKILL.md

在用户在第 2 轮选择的位置创建技能目录和文件。

使用此格式：

```markdown
---
name: {{skill-name}}
description: {{one-line description}}
allowed-tools:
  {{list of tool permission patterns observed during session}}
when_to_use: {{detailed description of when Claude should automatically invoke this skill, including trigger phrases and example user messages}}
argument-hint: "{{hint showing argument placeholders}}"
arguments:
  {{list of argument names}}
context: {{inline or fork -- omit for inline}}
---

# {{Skill Title}}
技能描述

## 输入
- `$arg_name`: 此输入的描述

## 目标
此工作流的明确目标。最好有明确定义的完成工件或标准。

## 步骤

### 1. 步骤名称
此步骤中要做什么。具体且可操作。在适当时包含命令。

**成功标准**：始终包含这个！这表明步骤已完成，我们可以继续。可以是列表。

重要提示：请参阅下面的每步注释部分，了解你可以为每步选择性包含的注释。

...
```

**每步注释**：
- **成功标准** 在每个步骤上是必需的。这有助于模型理解用户对其工作流的期望，以及何时应该有信心继续。
- **执行**：`Direct`（默认）、`Task agent`（直接子代理）、`Teammate`（具有真正并行性和代理间通信的代理）或 `[human]`（用户来做）。仅在非 Direct 时需要指定。
- **工件**：此步骤产生后续步骤需要的数据（例如，PR 编号、提交 SHA）。仅在后续步骤依赖它时包含。
- **人工检查点**：在继续之前何时暂停并询问用户。对于不可逆操作（合并、发送消息）、错误判断（合并冲突）或输出审查包含此项。
- **规则**：工作流的硬性规则。参考会话期间的用户纠正在此处特别有用。

**步骤结构提示：**
- 可以并发运行的步骤使用子编号：3a、3b
- 需要用户操作的步骤在标题中加 `[human]`
- 保持简单技能的简单——2 步技能不需要在每个步骤上都有注释

**前置说明规则：**
- `allowed-tools`：所需的最低权限（使用 `Bash(gh:*)` 等模式，而非 `Bash`）
- `context`：仅对不需要中途用户输入的独立技能设置 `context: fork`。
- `when_to_use` 至关重要——告诉模型何时自动调用。以"当...时使用"开始，包含触发短语。示例："当用户想要将 PR 精选到发布分支时使用。示例：'精选到发布'、'CP 这个 PR'、'热修复'。"
- `arguments` 和 `argument-hint`：仅在技能需要参数时包含。在正文中使用 `$name` 进行替换。

### 第 4 步：确认并保存

在写入文件之前，以 yaml 代码块的形式在你的响应中输出完整的 SKILL.md 内容，以便用户可以使用适当的语法高亮来审阅它。然后使用 AskUserQuestion 要求确认，提问简单，如"这个 SKILL.md 看起来可以保存吗？"——不要使用正文字段，保持问题简洁。

写入后，告诉用户：
- 技能保存在哪里
- 如何调用它：`/{{skill-name}} [arguments]`
- 他们可以直接编辑 SKILL.md 来细化它
