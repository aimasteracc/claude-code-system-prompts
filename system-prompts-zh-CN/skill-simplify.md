# 简化：代码审查与清理

审查所有已更改的文件，检查复用性、质量和效率。修复发现的任何问题。

## 第一阶段：识别更改

运行 `git diff`（如有已暂存的更改则运行 `git diff HEAD`）以查看更改内容。如果没有 git 更改，审查用户提到的或你在本次对话中早些时候编辑过的最近修改文件。

## 第二阶段：并行启动三个审查代理

使用 ${AGENT_TOOL_NAME} 工具在单条消息中同时启动所有三个代理。向每个代理传递完整的差异，以便其获得完整的上下文。

### 代理 1：代码复用审查

对于每项更改：

1. **搜索可以替代新编写代码的现有实用工具和辅助函数**。在代码库其他地方寻找相似模式——常见位置包括实用工具目录、共享模块以及与更改文件相邻的文件。
2. **标记任何复制了现有功能的新函数。** 建议使用已有的函数代替。
3. **标记任何可以使用现有实用工具的内联逻辑**——手写的字符串处理、手动路径处理、自定义环境检查、特定的类型守卫以及类似模式是常见候选。

### 代理 2：代码质量审查

审查相同的更改，查找不规范的模式：

1. **冗余状态**：复制现有状态的状态、可以派生的缓存值、可以替换为直接调用的观察者/效果
2. **参数膨胀**：向函数添加新参数而非概括化或重构现有参数
3. **略有不同的复制粘贴**：应该用共享抽象统一的近似重复代码块
4. **泄漏抽象**：暴露应该封装的内部细节，或破坏现有抽象边界
5. **字符串化类型**：在代码库中已有常量、枚举（字符串联合）或品牌类型的地方使用原始字符串
6. **不必要的 JSX 嵌套**：没有布局价值的包装 Box/元素——检查内部组件属性（flexShrink、alignItems 等）是否已提供所需行为
7. **不必要的注释**：解释代码做了什么的注释（良好命名的标识符已经做到了这点）、记录更改的注释、引用任务/调用者的注释——删除；仅保留非显而易见的原因（隐藏的约束、微妙的不变量、变通方案）

### 代理 3：效率审查

审查相同的更改，查找效率问题：

1. **不必要的工作**：冗余计算、重复的文件读取、重复的网络/API 调用、N+1 模式
2. **错过的并发机会**：顺序运行本可并行的独立操作
3. **热路径膨胀**：在启动或每次请求/渲染的热路径中添加新的阻塞工作
4. **周期性的空操作更新**：在无条件触发的轮询循环、定时器或事件处理器中进行状态/存储更新——添加变更检测守卫，避免在没有实际变化时通知下游消费者。另外：如果包装函数接受 updater/reducer 回调，验证它是否遵从同引用返回（或其他"无变化"信号）——否则调用者的提前返回空操作将被静默击败
5. **不必要的存在性检查**：在操作前检查文件/资源存在性（TOCTOU 反模式）——直接操作并处理错误
6. **内存问题**：无界数据结构、缺少清理、事件监听器泄漏
7. **过于宽泛的操作**：在只需要部分内容时读取整个文件，在只过滤一项时加载所有项目

## 第三阶段：修复问题

等待所有三个代理完成。汇总其发现并直接修复每个问题。如果某个发现是误报或不值得处理，记录并跳过——不要与发现争论，直接跳过即可。

完成后，简要总结修复了什么（或确认代码已经清晰）。
