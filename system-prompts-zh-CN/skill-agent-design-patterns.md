# 代理设计模式

本文涵盖在 Claude API 上构建代理的决策启发法：选择哪些原语、如何设计工具接口，以及如何在长时间运行中管理上下文和成本。有关每个工具的机制和代码示例，请参见 `tool-use-concepts.md` 和特定语言的文件夹。

---

## 模型参数

| 参数 | 使用场景 | 预期效果 |
| --- | --- | --- |
| **自适应思考**（`thinking: {type: "adaptive"}`） | 当你希望 Claude 控制何时以及思考多少时。 | Claude 根据每个请求确定思考深度，并在工具调用之间自动穿插思考。无需调整词元预算。 |
| **努力程度**（`output_config: {effort: ...}`） | 在彻底程度和词元效率之间调整权衡时。 | 较低的努力 → 更少且更合并的工具调用、更少的前言、更简短的确认。`medium` 通常是有利的平衡。当正确性比成本更重要时使用 `max`。 |

模型支持和参数详情见 `SKILL.md` §思考与努力。

## 设计你的工具接口

### Bash 与专用工具

Claude 不了解你的应用程序安全边界、审批策略或 UI 界面。Claude 发出工具调用；你的框架处理它们。这些工具调用的形状决定了框架能做什么。

**bash 工具**给了 Claude 广泛的编程杠杆——它几乎可以执行任何操作。但它只给框架一个不透明的命令字符串，每个操作的形状都相同。将操作提升为**专用工具**则给了框架一个带有类型参数的特定操作钩子，可以拦截、控制、呈现或审计。

**何时将操作提升为专用工具：**

- **安全边界。** 需要控制的操作是天然候选。可逆性是一个有用的标准：难以撤销的操作（外部 API 调用、发送消息、删除数据）可以在用户确认后才执行。`send_email` 工具很容易控制；`bash -c "curl -X POST ..."` 则不行。
- **陈旧检查。** 专用的 `edit` 工具可以在 Claude 上次读取文件后文件发生变化时拒绝写入。Bash 无法强制执行这个不变量。
- **渲染。** 某些操作受益于自定义 UI。Claude Code 将提问提升为工具，以便呈现为模态框、提供选项并阻塞代理循环直到得到回答。
- **调度。** 只读工具如 `glob` 和 `grep` 可以标记为并行安全。当相同操作通过 bash 运行时，框架无法区分并行安全的 `grep` 和不安全的 `git push`，因此必须顺序执行。

**经验法则：** 从 bash 开始以求广度。当需要控制、渲染、审计或并行化操作时，提升为专用工具。

## Anthropic 提供的工具

| 工具 | 执行端 | 使用场景 | 预期效果 |
| --- | --- | --- | --- |
| **Bash** | 客户端 | Claude 需要执行 shell 命令。 | Claude 发出命令；你的框架执行它们。提供了参考实现。 |
| **文本编辑器** | 客户端 | Claude 需要读取或编辑文件。 | Claude 通过你的实现查看、创建和编辑文件。提供了参考实现。 |
| **Computer use** | 客户端或服务器端 | Claude 需要与 GUI、Web 应用或视觉界面交互。 | Claude 截图并发出鼠标/键盘命令。可以自托管（你运行环境）或 Anthropic 托管。 |
| **代码执行** | 服务器端 | Claude 需要在你不想管理的沙箱中运行代码。 | Anthropic 托管的容器，内置文件和 bash 子工具。无需客户端执行。 |
| **网络搜索/获取** | 服务器端 | Claude 需要训练截止日期之后的信息（新闻、当前事件、最新文档）或特定 URL 的内容。 | Claude 发出查询或 URL；Anthropic 执行并返回带引用的结果。 |
| **记忆** | 客户端 | Claude 需要在会话间保存上下文。 | Claude 读写 `/memories` 目录。你实现存储后端。 |

**客户端**工具由 Anthropic 定义（名称、模式、Claude 的使用模式），但由你的框架执行。Anthropic 提供参考实现。**服务器端**工具完全在 Anthropic 基础设施上运行——在 `tools` 中声明，Claude 自动处理其余部分。

## 组合工具调用：编程式工具调用

使用标准工具调用时，每次工具调用都是一次往返：Claude 调用工具，结果进入 Claude 的上下文，Claude 推理后再调用下一个工具。三个顺序操作（读取资料 → 查找订单 → 检查库存）意味着三次往返。每次都增加延迟和词元，而大部分中间数据再也不需要了。

**编程式工具调用（PTC）**让 Claude 将这些调用组合成一个脚本。脚本在代码执行容器中运行。当脚本调用工具时，容器暂停，调用被执行（客户端或服务器端），结果返回到正在运行的代码——而不是 Claude 的上下文。脚本用正常的控制流（循环、过滤、分支）处理结果。只有脚本的最终输出返回给 Claude。

| 使用场景 | 预期效果 |
| --- | --- |
| 许多顺序工具调用，或者你希望在大型中间结果进入上下文窗口之前过滤它们。 | Claude 编写以函数形式调用工具的代码。在代码执行容器中运行。词元成本与最终输出成比例，而非与中间结果。 |

## 扩展工具和指令集

| 功能 | 使用场景 | 预期效果 |
| --- | --- | --- |
| **工具搜索** | 可用工具很多，但每个请求只有少数相关。不想一开始就把所有模式放入上下文。 | Claude 搜索工具集，只加载相关模式。工具定义被追加而非替换——保留缓存（见下文缓存部分）。 |
| **技能** | 特定任务的指令，Claude 只有在相关时才加载。 | 每个技能是一个带有 `SKILL.md` 的文件夹。技能的描述默认在上下文中；任务需要时 Claude 读取完整文件。 |

这两种模式都使固定上下文保持较小，并按需加载细节。

## 长时间运行的代理：管理上下文

| 模式 | 使用场景 | 预期效果 |
| --- | --- | --- |
| **上下文编辑** | 上下文在多个轮次后变得陈旧（旧的工具结果、已完成的思考）。 | 根据可配置的阈值清除工具结果和思考块。保持对话简洁而无需摘要。 |
| **压缩** | 对话可能达到或超过上下文窗口限制。 | 较早的上下文在服务器端被摘要为压缩块。有关关键的 `response.content` 处理，见 `SKILL.md` §压缩。 |
| **记忆** | 状态必须跨会话持久（不仅限于一次对话内）。 | Claude 读写记忆目录中的文件。在进程重启后依然存在。 |

**如何选择：** 上下文编辑和压缩在会话内操作——编辑修剪陈旧的轮次，压缩在接近限制时进行摘要。记忆用于跨会话持久性。许多长时间运行的代理同时使用三者。

## 代理的缓存

**首先阅读 `prompt-caching.md`。** 它涵盖前缀匹配不变量、断点放置、静默失效审计，以及为什么在会话中更改工具或模型会破坏缓存。本节仅涵盖这些约束的代理特定变通方法。

| 约束（来自 `prompt-caching.md`）| 代理特定变通方法 |
| --- | --- |
| 会话中编辑系统提示词使缓存失效。 | 改为在 `messages` 数组中追加 `<system-reminder>` 块。缓存前缀保持完整。Claude Code 用此方法处理时间更新和模式切换。 |
| 会话中切换模型使缓存失效。 | 用较便宜的模型为子任务派生**子代理**；在一个模型上保持主循环。Claude Code 的 Explore 子代理就是这样使用 Haiku 的。 |
| 会话中添加/删除工具使缓存失效。 | 使用**工具搜索**进行动态发现——它追加工具模式而非替换它们，从而保留现有前缀。 |

关于多轮断点放置，使用顶层自动缓存——见 `prompt-caching.md` §放置模式。
