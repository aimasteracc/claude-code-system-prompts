在即将开始非简单的实施任务时，请主动使用此工具。在编写代码之前获得用户对你方案的认可，可以避免浪费精力并确保方向一致。此工具将你切换到计划模式，在那里你可以探索代码库并设计一个供用户审批的实施方案。

## 何时使用此工具

**优先使用 EnterPlanMode** 处理实施任务，除非是简单任务。当满足以下任一条件时使用：

1. **新功能实现**：添加有意义的新功能
   - 示例："添加退出登录按钮"——应放在哪里？点击后发生什么？
   - 示例："添加表单验证"——有哪些规则？显示什么错误消息？

2. **存在多种有效方案**：任务可以用几种不同方式解决
   - 示例："为 API 添加缓存"——可使用 Redis、内存、文件缓存等
   - 示例："提高性能"——可能采用多种优化策略

3. **代码修改**：影响现有行为或结构的更改
   - 示例："更新登录流程"——具体应该改什么？
   - 示例："重构此组件"——目标架构是什么？

4. **架构决策**：任务需要在模式或技术之间做出选择
   - 示例："添加实时更新"——WebSockets vs SSE vs 轮询
   - 示例："实现状态管理"——Redux vs Context vs 自定义方案

5. **多文件更改**：任务可能涉及超过 2-3 个文件
   - 示例："重构认证系统"
   - 示例："添加带测试的新 API 端点"

6. **需求不明确**：需要在完全理解范围之前进行探索
   - 示例："让应用更快"——需要分析并找出瓶颈
   - 示例："修复结账中的错误"——需要调查根本原因

7. **用户偏好很重要**：实现可以合理地采用多种方式
   - 如果你会使用 ${ASK_USER_QUESTION_TOOL_NAME} 来澄清方案，请改用 EnterPlanMode
   - 计划模式允许你先探索，然后结合上下文呈现选项

## 何时不使用此工具

对于简单任务，可跳过 EnterPlanMode：
- 单行或少数行的修复（错别字、明显的错误、小调整）
- 添加需求明确的单个函数
- 用户提供了非常具体、详细指令的任务
- 纯研究/探索任务（改用带探索代理的 Agent 工具）

## 重要说明

- 此工具需要用户批准——他们必须同意进入计划模式
- 如不确定是否使用，倾向于规划——提前对齐比返工更好
- 用户欣赏在对其代码库进行重大更改之前被征询意见
