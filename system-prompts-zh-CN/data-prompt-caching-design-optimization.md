# 提示缓存 — 设计与优化

本文件介绍如何设计提示构建代码以实现有效缓存。语言特定的语法请参见各语言 README 或单文件文档中的 `## Prompt Caching` 章节。

## 一切都源于这一不变的原则

**提示缓存是前缀匹配。前缀中任何位置的任何变化都会使其后的所有内容失效。**

缓存键由每个 `cache_control` 断点处渲染提示的精确字节派生。位置 N 处单字节的差异 — 时间戳、重排序的 JSON 键、列表中不同的工具 — 都会使位置 ≥ N 处的所有断点缓存失效。

渲染顺序为：`tools` → `system` → `messages`。最后一个 system 块上的断点会将 tools 和 system 一起缓存。

围绕这一约束设计提示构建路径。排序正确时，大多数缓存自然生效。排序错误时，无论添加多少 `cache_control` 标记都无济于事。

---

## 优化现有代码的工作流程

当被要求添加或优化缓存时：

1. **追踪提示组装路径。** 找出 `system`、`tools` 和 `messages` 的构建位置。识别流入其中的每个输入。
2. **按稳定性对每个输入分类：**
   - 从不变化 → 放在提示早期，在任何断点之前
   - 每个会话变化 → 放在全局前缀之后，按会话缓存
   - 每轮变化 → 放在末尾，在最后一个断点之后
   - 每次请求变化（时间戳、UUID、随机 ID）→ **消除或移至最末尾**
3. **检查渲染顺序是否与稳定性顺序匹配。** 稳定内容必须物理上先于可变内容。如果时间戳被插值进系统提示头部，则其后的所有内容都无法缓存，无论添加了什么标记。
4. **在稳定性边界处放置断点。** 参见下面的放置模式。
5. **审计静默失效因素。** 参见反模式表。

---

## 放置模式

### 跨多个请求共享的大型系统提示

在最后一个 system 文本块上放置断点。如果有工具，它们在 system 之前渲染 — 最后一个 system 块上的标记会将 tools + system 一起缓存。

```json
"system": [
  {"type": "text", "text": "<large shared prompt>", "cache_control": {"type": "ephemeral"}}
]
```

### 多轮对话

在最新追加轮次的最后一个内容块上放置断点。后续每次请求都会复用整个先前对话前缀。早期断点仍作为有效读取点，随着对话增长，命中次数会逐步累积。

```json
// 最后一个用户轮次的最后一个内容块
messages[-1].content[-1].cache_control = {"type": "ephemeral"}
```

### 共享前缀，变化后缀

许多请求共享大型固定序言（少样本示例、检索到的文档、说明），但最终问题不同。将断点放在**共享**部分的末尾，而非整个提示的末尾 — 否则每次请求都会写入不同的缓存条目，永远不会有读取命中。

```json
"messages": [{"role": "user", "content": [
  {"type": "text", "text": "<shared context>", "cache_control": {"type": "ephemeral"}},
  {"type": "text", "text": "<varying question>"}  // 不添加标记 — 每次都不同
]}]
```

### 每次都从头变化的提示

不要缓存。如果前 1K 个词元每次请求都不同，就没有可复用的前缀。添加 `cache_control` 只会产生缓存写入溢价而零读取收益。省略它。

## 架构指导

这些决策比标记放置更重要。优先修复这些问题。

**保持系统提示冻结。** 不要在系统提示中插值"当前日期：X"、"模式：Y"、"用户名：Z" — 这些内容位于前缀的开头，会使其后的所有内容失效。将动态上下文作为用户或助手消息注入 `messages` 的后面。第 5 轮的消息不会使第 5 轮之前的任何内容失效。

**不要在对话中途更换工具或模型。** 工具在位置 0 渲染；添加、删除或重排序工具会使整个缓存失效。切换模型也是如此（缓存是模型范围的）。如果需要"模式"，不要替换工具集 — 给 Claude 一个记录模式转换的工具，或将模式作为消息内容传入。按名称对工具进行确定性排序。

**分叉操作必须复用父级的精确前缀。** 侧面计算（摘要、压缩、子代理）通常会启动单独的 API 调用。如果分叉以任何差异重建 `system` / `tools` / `model`，它将完全错过父级的缓存。逐字复制父级的 `system`、`tools` 和 `model`，然后在末尾追加分叉特定的内容。

## 静默失效因素

审查代码时，在任何流入提示前缀的内容中搜索这些模式：

| 模式 | 破坏缓存的原因 |
|---|---|
| `datetime.now()` / `Date.now()` / `time.time()` 在系统提示中 | 前缀每次请求都会变化 |
| `uuid4()` / `crypto.randomUUID()` / 请求 ID 在内容早期 | 同上 — 每次请求都是唯一的 |
| `json.dumps(d)` 未使用 `sort_keys=True` / 迭代 `set` | 非确定性序列化 → 前缀字节不同 |
| f-string 将会话/用户 ID 插值到系统提示中 | 每用户前缀不同；不跨用户共享 |
| 条件系统段（`if flag: system += ...`） | 每种标志组合都是不同的前缀 |
| `tools=build_tools(user)` 集合因用户而异 | 工具在位置 0 渲染；跨用户无缓存 |

修复方法：将动态部分移到最后一个断点之后、使其确定性，或如果不是必要内容则删除。

## API 参考

```json
"cache_control": {"type": "ephemeral"}              // 5 分钟 TTL（默认）
"cache_control": {"type": "ephemeral", "ttl": "1h"} // 1 小时 TTL
```

- 每次请求最多 **4** 个 `cache_control` 断点。
- 可放在任意内容块上：system 文本块、工具定义、消息内容块（`text`、`image`、`tool_use`、`tool_result`、`document`）。
- `messages.create()` 上的顶层 `cache_control` 自动放置在最后一个可缓存块上 — 不需要精细放置时最简单的选择。
- 最小可缓存前缀取决于模型。较短的前缀即使有标记也会静默地不被缓存 — 不会报错，只是 `cache_creation_input_tokens: 0`：

| 模型 | 最小值 |
|---|---:|
| Opus 4.6, Opus 4.5, Haiku 4.5 | 4096 词元 |
| Sonnet 4.6, Haiku 3.5, Haiku 3 | 2048 词元 |
| Sonnet 4.5, Sonnet 4.1, Sonnet 4, Sonnet 3.7 | 1024 词元 |

3K 词元的提示在 Sonnet 4.5 上可以缓存，但在 Opus 4.6 上会静默地不被缓存。

**经济学：** 缓存读取费用约为基础输入价格的 0.1 倍。缓存写入费用为 **5 分钟 TTL 时 1.25 倍，1 小时 TTL 时 2 倍**。盈亏平衡取决于 TTL：5 分钟 TTL 时，两次请求即可盈亏平衡（1.25× + 0.1× = 1.35× vs 2× 未缓存）；1 小时 TTL 时，至少需要三次请求（2× + 0.2× = 2.2× vs 3× 未缓存）。1 小时 TTL 可在突发流量的间隔期间保持条目有效，但写入费用加倍意味着需要更多读取才能回本。

## 验证缓存命中

响应的 `usage` 对象报告缓存活动：

| 字段 | 含义 |
|---|---|
| `cache_creation_input_tokens` | 本次请求写入缓存的词元数（您支付了约 1.25 倍写入溢价） |
| `cache_read_input_tokens` | 本次请求从缓存读取的词元数（您支付了约 0.1 倍） |
| `input_tokens` | 按全价处理的词元数（未缓存） |

如果在具有相同前缀的重复请求中 `cache_read_input_tokens` 始终为零，则存在静默失效因素 — 对两次请求的渲染提示字节进行差异比较以找到问题所在。

**`input_tokens` 仅是未缓存的剩余部分。** 提示总大小 = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`。如果代理运行了数小时但 `input_tokens` 显示 4K，则其余部分来自缓存 — 检查总和，而非单个字段。

各语言的访问方式：`response.usage.cache_read_input_tokens`（Python/TS/Ruby）、`$message->usage->cacheReadInputTokens`（PHP）、`resp.Usage.CacheReadInputTokens`（Go/C#）、`.usage().cacheReadInputTokens()`（Java）。

## 失效层级

并非每次参数变化都会使所有内容失效。API 有三个缓存层，变化只会使其自身所在层及以下层失效：

| 变化 | 工具缓存 | 系统缓存 | 消息缓存 |
|---|:---:|:---:|:---:|
| 工具定义（添加/删除/重排序） | ❌ | ❌ | ❌ |
| 切换模型 | ❌ | ❌ | ❌ |
| `speed`、网络搜索、引用开关 | ✅ | ❌ | ❌ |
| 系统提示内容 | ✅ | ❌ | ❌ |
| `tool_choice`、图像、`thinking` 启用/禁用 | ✅ | ✅ | ❌ |
| 消息内容 | ✅ | ✅ | ❌ |

含义：可以在每次请求中更改 `tool_choice` 或切换 `thinking`，而不会丢失 tools+system 缓存。无需过度担心这些 — 只有工具定义和模型更改才会强制完全重建。

---

## 20 块回溯窗口

每个断点向后**最多 20 个内容块**查找先前的缓存条目。如果单个轮次添加了超过 20 个块（在包含大量 tool_use/tool_result 对的代理循环中很常见），下一次请求的断点将无法找到先前的缓存，会静默地错过。

修复方法：在长轮次中每约 15 个块放置一个中间断点，或将标记放置在距上一轮次最后一个缓存块 20 个以内的块上。

---

## 并发请求时序

缓存条目只有在第一个响应**开始流式传输**后才可读取。N 个具有相同前缀的并行请求都会按全价付费 — 没有请求可以读取其他请求仍在写入的内容。

对于扇出模式：发送 1 个请求，等待第一个流式词元（而非完整响应），然后发出剩余 N-1 个请求。它们将读取第一个请求刚写入的缓存。
