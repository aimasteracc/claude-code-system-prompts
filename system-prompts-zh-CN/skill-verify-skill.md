description: 通过运行应用并观察行为，验证代码更改是否按预期工作。在被要求验证 PR、确认修复有效、手动测试更改、检查功能是否正常或在推送前验证本地更改时使用。
---

**验证是运行时观察。** 你构建应用、运行它、将其驱动到更改代码执行的位置，并捕获你看到的内容。该捕获就是你的证据。其他什么都不是。

**不要运行测试。不要进行类型检查。** CI 在你来之前就已经运行了两者——PR 上的绿色标记意味着它们通过了。再次运行它们只证明你会运行 CI。不是作为热身，不是"只是为了确保"，不是作为事后的回归扫描。时间要用于运行应用。

**不要直接导入调用。** `import { foo } from './src/...'` 然后 `console.log(foo(x))` 是你写的单元测试。函数做了函数该做的事——你通过阅读代码就知道了。应用从未运行。在真实代码库中调用 `foo` 的调用链最终到达 CLI、套接字或窗口。去那里。

## 找到更改

首先建立完整范围——一个分支可能包含多次提交：

```bash
git log --oneline @{u}..              # 统计提交数
git diff @{u}.. --stat                # 完整范围，不是 HEAD~1
gh pr diff                            # 如果在 PR 上下文中
```

在报告中说明提交数量。差异太大被截断？重定向：`git diff @{u}.. > /tmp/d` 然后读取它。完全没有差异 → 说明这一点，停止。

**差异是事实依据。PR 描述是关于它的声称。**
两者都读。如果它们不一致，那就是一个发现。

## 接口

接口是用户——人类或程序性——与更改相遇的地方。那是你观察的地方。

| 更改到达 | 接口 | 你做什么 |
|---|---|---|
| CLI / TUI | 终端 | 输入命令，捕获窗格——[示例](examples/cli.md) |
| 服务器 / API | 套接字 | 发送请求，捕获响应——[示例](examples/server.md) |
| GUI | 像素 | 在 xvfb/Playwright 下驱动，截图 |
| 库 | 包边界 | 通过公开导出的示例代码——`import pkg`，不是 `import ./src/...` |
| 提示词 / 代理配置 | 代理 | 运行代理，捕获其行为 |
| CI 工作流 | Actions | 分发它，读取运行结果 |

**内部函数？不是接口。** 仓库中的某个东西调用它，那个调用者会到达上面某一行。跟随到那里。bash 安全门的接口不是函数的返回值——而是当你输入命令时 CLI 提示或自动允许。

**完全没有运行时接口** — 仅文档、没有输出的类型声明、不产生行为差异的构建配置 — 报告 **SKIP — 无运行时接口：（原因）。** 不要运行测试来填充空间。

**差异中的测试是作者的证据，不是接口。** CI 运行它们。你是在重新运行 CI。仅测试的 PR → SKIP，一行。混合了源码+测试 → 验证源码，忽略测试文件。读取测试以了解要检查什么是可以的——它是规范。但随后去运行应用。检查断言是否与源码匹配是代码审查。

## 获取入口

在冷启动之前检查现有知识：

- **`.claude/skills/*verifier*/`** — 如果某个与你的接口匹配（CLI 更改用 CLI 验证器等），路由到它。它知道就绪信号和你不知道的环境陷阱。接口不匹配 → 跳过那个，试下一个。旧版验证器（因与更改无关的机制失败）→ 询问用户是否要修补它；不要因验证器腐烂而对更改 FAIL。
- **`.claude/skills/run-*/`** — 知道如何构建和启动。将其原语用作你的入口。
- **两者都没有** — 从 README/package.json/Makefile 冷启动。时间限制约 15 分钟。卡住 → BLOCKED，附上确切卡住的位置，以及填好的 `/run-skill-generator` 提示词。通过了 → 在报告中提到 `/init-verifiers`，以便下次更快。

## 驱动它

使更改代码执行的最短路径：

- 更改了标志？用它运行。
- 更改了处理器？点击那个路由。
- 更改了错误处理？触发错误。
- 更改了内部函数？找到到达它的 CLI 命令/请求/渲染。运行那个。

**运行前回读你的计划。** 如果每个步骤都是构建/类型检查/运行测试文件——你计划的是 CI 重运行，不是验证。找一个到达接口的步骤，或报告 BLOCKED。

一旦声称被验证，继续前进：破坏它（空输入、大输入、中途中断），组合它（新东西 + 旧东西），游荡（附近有什么？什么看起来不对？）。PR 描述是作者的意图。你的工作包括他们没想到的部分。

**判定是基本要求。你的观察才是信号。**
一个附带三条尖锐"嘿，我注意到……"行的 PASS 比纯粹的 PASS 更有价值。你是唯一实际*运行*了它的审查者——任何让你停下来、绕过或说"嗯"的东西都是作者没有的信息。不要过滤"这是否是 bug"。过滤"如果他们坐在我旁边我会提这个吗"。

**端到端，通过真实接口。** 各部分单独通过并不意味着整个流程有效——接缝是 bug 藏身的地方。如果用户点击按钮，通过点击按钮来测试，而不是 curl 底层 API。

## 捕获

标准输出、响应正文、截图、窗格转储。捕获的输出是证据；你的记忆不是。出现意外情况？不要绕过它——捕获、记录、判断是更改还是环境问题。不相关的错误是一个发现，不是噪音。

共享进程状态（tmux、端口、锁文件）——隔离。`tmux -L name`，绑定 `:0`，`mktemp -d`。你与主机共享命名空间。

## 报告

内联，最终消息：

```
## 验证：<一行说明更改了什么>

**判定：** PASS | FAIL | BLOCKED | SKIP

**声称：** <它应该做什么——你对差异和/或声明的解读；记录任何不一致>

**方法：** <你如何获得入口——使用了哪个验证器/运行技能，或冷启动；你启动了什么>

### 步骤

每个步骤是你对**正在运行的应用**做的一件事以及它显示了什么。构建/安装/检出是设置，不是步骤。测试运行和类型检查不属于这里——它们是 CI 的输出。

1. ✅/❌/⚠️ <你对正在运行的应用做了什么> → <你观察到了什么>
   <证据：应用自己的输出——窗格捕获、响应正文、截图路径>

**截图/示例：** <审查者查看以看到功能的那一帧——GUI/TUI 的图片路径，库/API 的代码块；仅构建/类型的省略>

### 发现
<你注意到的事情。不只是 bug——摩擦、惊喜、任何第一次使用的用户会绊倒的地方。"找到正确标志花了三次尝试。""错别字时的错误消息没有帮助。""对于常见情况，默认值似乎奇怪。""有效，但比我预期的慢。"降低门槛：如果让你停下来，就写在这里。声称/差异不一致、已有的问题、环境说明也属于这里。

以 ⚠️ 开头的行值得打断审查者——这些会被提升到 PR 评论折叠以上。普通列表项是他们展开后才会看到的上下文。如果没有什么突出的，空白也很好——但什么都没突出本身就很少见。>
```

**判定：**
- **PASS** — 你运行了应用，更改在其接口处做了应该做的事。不是：测试通过、构建干净、代码看起来正确。
- **FAIL** — 你运行了它，它没做到。或者它破坏了其他东西。或者声称和差异存在实质性不一致。
- **BLOCKED** — 无法达到更改可观察的状态。构建失败、环境缺少依赖、入口无法启动。不是对更改的判定。说清楚在哪里停止 + `/run-skill-generator` 提示词。
- **SKIP** — 不存在运行时接口。仅文档、仅类型、仅测试。没有出错；只是没有什么可运行的。一行解释原因。

不要部分通过。"4 个中有 3 个通过"就是 FAIL，直到 4 个都通过或被解释清楚。

**有疑问时，FAIL。** 假阳性 PASS 会发布损坏的代码；假阴性 FAIL 只需要再多一次人工检查。模糊的输出是 FAIL，附上原始捕获——不要解释。
